# Лабораторна робота №5 — Варіант 1 (Події та журнал змін)

Коротко — що реалізовано:

- Делегат `StudentListHandler` і клас подій `StudentListHandlerEventArgs` для передачі інформації про зміну.
- `StudentCollection` (в `src/Collections/StudentCollection.cs`) тепер має:
  - властивість `Name`
  - події `StudentCountChanged` та `StudentReferenceChanged`
  - індексатор `this[int index]` для заміни елементів
  - метод `Remove(int index)` для видалення студента з повідомленням події
- `Journal` і `JournalEntry` (в `src/Collections/Journal.cs` та `src/Collections/JournalEntry.cs`) — накопичують записи про події та надають простий вивід у консоль.
- Програма в `src/Program.cs` демонструє підписку журналів на події колекцій і виконання операцій (додавання, видалення, заміна).

Файли

- `src/Events/StudentListHandlerEventArgs.cs`
- `src/Collections/StudentCollection.cs`
- `src/Collections/Journal.cs`
- `src/Collections/JournalEntry.cs`
- `src/Program.cs` (демонстрація ЛР5)

Як запустити 

1. З командного рядка у корені проекту виконайте:

```powershell
dotnet build
dotnet run --project Variant1.csproj
```

2. У консолі буде виконано кілька демонстраційних сценаріїв. Для ЛР5 програма:
  - створює дві колекції `StudentCollection` та два `Journal`;
  - підписує журнали на події обох колекцій;
  - виконує операції: `AddDefaults`, `AddStudents`, `Remove`, заміну через індексатор;
  - виводить вміст журналів (послідовність `JournalEntry`) у консоль.

Приклад очікуваного виводу:

```
Journal 1 entries:
- [CollectionA] Count changed: +10 (now 10)
- [CollectionA] Student at index 3 replaced: Ivanov -> Petrenko
- [CollectionA] Student removed at index 2: Sidorov

Journal 2 entries:
- [CollectionB] Count changed: +5 (now 5)
- [CollectionB] Student added: Shevchenko
```
 
Теоретичні відомості (ЛР5)

- Події й делегати: у C# делегат — це тип, що описує підпис методу; подія — захищений механізм публікації повідомлень, який використовує делегати. Події застосовуються для реалізації патерну "спостерігач" (observer).
- Класи подій: `EventArgs` або власні класи-наслідники (тут `StudentListHandlerEventArgs`) переносять додаткові дані про зміну (ім'я колекції, індекс, опис зміни, дані студента).
- Розділення відповідальностей: колекція генерує події, а журнал/логер підписується та записує інформацію — це забезпечує слабке зв'язування модулів.
- Синхронність і продуктивність: виклик подій за замовчуванням синхронний; у великих операціях (масове додавання, імпорт) рекомендується емінувати агреговану подію (batch) або виконувати логування асинхронно, щоб не блокувати потокороботу.
- Атомарність і безпека потоків: при багатопотоковому доступі потрібно захищати колекцію та безпечно викликати підписники (копіювати посилання на делегат перед інвокомом). Для реальних застосунків рекомендується використовувати блокування або потокобезпечні колекції.
- Журнал і збереження: `Journal` тут зберігає записи в пам'яті; для довготривалих дослідів або аналізу рекомендується експортувати у CSV/JSON або записувати в базу/файл, щоб уникнути надмірного споживання пам'яті.
- Тестування: пишіть юніт-тести, що перевіряють, що події викликаються з правильними `EventArgs` і вірними значеннями після операцій Add/Remove/Replace.
- Практичне застосування: подібна схема корисна для UI-повідомлень, аудиту змін даних, збереження історії редагувань і реалізації механізмів "undo/redo".
